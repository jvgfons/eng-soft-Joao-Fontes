## Contextualização

As mudanças rápidas que ocorrem em diversos setores das mais variadas empresas do mundo fizeram surgir os Métodos Ágeis na Engenharia de Software, na década de 90, a maioria das operações já dependiam de softwares, para atender suas demandas e pressionados pela concorrência, as empresas estavam dispostas até a comprometer alguns requisitos para ter um novo software rapidamente.
Junto com a mudança no cenário das empresas, os requisitos para software mudavam também, então era necessário o desenvolvimento rápido e capaz de lidar com essas mudanças, isso decolou no final dos anos 90 com a criação do Extreme Programming, Scrum e o DSDM

## Técnica ou Método de Desenvolvimento Ágil XP

A ideia da Programação Extrema (XP), é uma técnica de desenvolvimento ágil, a ideia seria levar a níveis extremos as boas práticas reconhecidas, nela, um mesmo programa pode ter várias releases lançadas por diferentes programadores, isso ocorre em um ciclo, eles selecionam as histórias de usuários, decompõe elas em tarefas, planeja o lançamento para uma data próxima, desenvolvem, integram com outras versões e testam, lançam a versão e avaliam. Na XP o desenvolvimento era apoiado pelo cliente em diversas interações onde ele é um dos testadores do sistema enquanto está sendo desenvolvido, as pessoas são apoiadas e não os processos. A XP era um pouco difícil de ser implementada em não se adequada de imediato a cultura da maioria das empresas.

## Desenvolvimento Ágil x Dirigido por Plano

Todos os métodos ágeis têm características em comum, o documento de requisitos é uma definição mais resumida, que contém apenas as características principais do sistema, diferente do software diferido por plano, como por exemplo um software programado para controlar uma aeronave, que deveriam ter os requisitos muito bem definidos, o sistema é desenvolvido em incrementos, por exemplo, em uma aplicação de software similar ao Paint do Windows, a empresa responsável entrega primeiro a tela com o quadro branco, depois o menu de seleção de cores, depois a ferramenta linha, e assim por diante, a cada entrega eles entram em contato com o cliente para verificar se está ok ou se algo precisa mudar, dessa forma é possível adaptar as mudanças de requisitos, até mesmo incluir ou remover requisitos do software.



## Histórias de Usuário x Engenharia de Requisitos

As histórias de usuário, são basicamente, um texto contendo o que o usuário quer fazer no sistema, o que ele quer ver na interface, em que ele quer clicar ou digitar, ou outras interações, e o que ele espera que o software faça com essas informações, conversando com o cliente sobre as histórias de usuário, os desenvolvedores conseguem entender quais delas são prioridades e que possam ser implementadas rapidamente, para que o cliente teste e gere o feedback, como estamos falando de um cenário em que os requisitos mudam, é natural que as histórias de usuários mudem durante o desenvolvimento.
Essa ferramenta é muito útil para o desenvolvimento pois é mais fácil trabalhar com elas do que com um documento convencional de requisitos ou casos de uso, porém pode acontecer de essas histórias não cobrirem todos os requisitos do sistema, o que pode gerar algum bug entre interações dentro do sistema por falta de alguma informação.
